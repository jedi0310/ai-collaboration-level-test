# AI 协同等级测试网站设计计划

## 背景

用户希望基于文章《观察了三年，我把所有人用AI的水平分成了10个等级。》做一个可部署的网站。访问者通过 6 到 8 个问题完成测试，得到一份个性化报告，知道自己目前处在 AI 使用能力的哪个等级，以及下一步更适合尝试哪种 AI 协同工作方式。

项目目标是先做一个方便部署、容易传播的网站。网站前端优先部署到 GitHub Pages，不做登录；公开测试版通过 Cloudflare Worker 和 D1 保存测试结果与报告，用于站长改进测试体验。

## 核心模型

文章中的 10 个等级应保留为报告锚点，因为“你在第几级”本身具有传播感和记忆点。

等级包括：

- Lv.0 旁观者
- Lv.1 尝鲜者
- Lv.2 对话者
- Lv.3 驯化师
- Lv.4 越境者
- Lv.5 织网者
- Lv.6 召唤师
- Lv.7 铸造师
- Lv.8 造物主
- Lv.9 觉醒者
- Lv.10 一人军团

测试不只看总分，而是围绕四个维度判断：

- 可控性：能否约束 AI、喂上下文、拆任务、控制输出质量。
- 广度：是否跨出本专业，把 AI 用到更多领域。
- 形态：是否从 ChatBot 对话进入 Agent、工具调用和长程任务。
- 角色：是否从消费者变成 Prompt、模板、知识库、Skill、工作流的创造者。

## 产品形态

网站包含三个核心页面或状态：

1. 开始页
   - 说明这是一个 AI 协同能力测试。
   - 强调测试结果不是评判，而是给出下一步升级路线。
   - 提供开始测试按钮。

2. 用户画像页
   - 收集用户名字/网名、行业、职位和手机号。
   - 名字/网名、行业必填；职位和手机号选填。
   - 用于让 DeepSeek 生成更贴近个人处境的报告。
   - 不做登录；公开测试版会通过 Cloudflare Worker 保存画像、选填联系方式、答题结果和报告。

3. 答题页
   - 8 道单选题。
   - 每题 4 个选项。
   - 移动端优先，适合在微信内打开。
   - 答题过程中显示进度，不展示复杂分数。

4. 报告页
   - 展示当前等级。
   - 展示四维能力画像。
   - 展示个性化分析报告。
   - 展示下一步推荐的 AI 协同方式。
   - 提供重新测试、下载报告的入口。

## 题目方向

建议第一版做 8 道题：

1. 你现在使用 AI 的频率如何？
2. AI 答得不好时，你通常怎么处理？
3. 你是否会给 AI 背景、案例、输出格式和限制条件？
4. 你使用 AI 的场景有多宽？
5. AI 是否已经进入你的固定工作流程？
6. 你是否使用过 Agent、Cursor、Claude Code、Codex、MCP 或自动化工具？
7. 你是否沉淀过自己的 Prompt、模板、知识库、Skill 或工作流？
8. 遇到新任务时，你第一反应是自己做，还是设计一套人机协作流程？

每个选项同时影响总分和四个维度分。题目不直接问“你是几级”，而是通过行为证据判断。

公测版展示时，每题选项会在一次测试开始时稳定乱序，避免用户通过 A/B/C/D 判断分值高低。评分仍以选项自身分值为准，不依赖展示位置。

## 评分逻辑

前端负责结构化诊断，不把最终报告完全交给模型自由判断。

前端计算：

- 初步等级：Lv.0 到 Lv.10。
- 四维分数：可控性、广度、形态、角色。
- 行为证据：从用户选项中提炼出的关键描述。
- 当前瓶颈：最弱维度或关键门槛。
- 下一步突破方向：例如从 Prompt 到流程、从 ChatBot 到 Agent、从使用到创造。

等级判断采用“总分 + 关键门槛”：

- 没真正使用过 AI，直接判定 Lv.0。
- 只会简单问答，通常不超过 Lv.1。
- 会追问和补背景，可进入 Lv.2。
- 会结构化 Prompt、约束输出、分步迭代，可进入 Lv.3。
- 开始跨领域使用，可进入 Lv.4。
- 有固定流程、模板、项目或知识库，可进入 Lv.5。
- 使用 Agent 或工具调用，可进入 Lv.6。
- 创造 Skill、Agent、自动化流程，可进入 Lv.7。
- AI 深度参与工作和创作，可进入 Lv.8。
- AI 成为思维方式和创造方法论，可进入 Lv.9。
- 构建一套个人 AI 系统，并能跨多个领域稳定输出，可进入 Lv.10。

这样可以减少虚高。例如用户分数很高，但从未使用 Agent，就不应直接给到 Lv.7。

## DeepSeek 报告生成

报告生成采用“前端诊断 + DeepSeek 表达”的方式。

前端传给模型的是结构化结果，而不是原始答案。示例：

```json
{
  "level": "Lv.5 织网者",
  "profile": {
    "name": "Jedi",
    "industry": "内容创作",
    "role": "创始人"
  },
  "scores": {
    "control": 7,
    "breadth": 6,
    "form": 4,
    "role": 5
  },
  "evidence": [
    "会给 AI 背景和输出格式",
    "已经有固定 Prompt 模板",
    "还没有稳定使用 Agent",
    "主要在内容和运营场景使用"
  ],
  "bottleneck": "形态维度偏弱，仍主要停留在 ChatBot 协作",
  "next_breakthrough": "从 ChatBot 协作进入 Agent 协作",
  "style": "中文、直接、鼓励、具体、有一点游戏升级感"
}
```

DeepSeek 输出结构化报告，建议包含：

- title：报告标题
- level_summary：当前等级一句话画像
- profile：个性化画像
- dimension_analysis：四维能力分析
- bottleneck：当前最容易卡住的地方
- next_steps：3 条下一步建议
- collaboration_modes：推荐尝试的 2 到 3 种 AI 协同方式
- closing：一句行动建议

报告风格应固定为：

- 中文表达。
- 直接、可执行。
- 鼓励但不夸张。
- 不制造焦虑。
- 保留文章里的游戏升级感。
- 少讲概念，多告诉用户下一步怎么做。

## API 与部署边界

纯前端网页不能安全内置 DeepSeek API Key。如果把 API Key 写进 GitHub Pages 前端代码，任何访问者都可以在浏览器里看到。

因此第一版支持两个模式：

1. 纯静态无后端模式
   - 部署到 GitHub Pages。
   - 页面不内置 API Key。
   - 用户可以手动填入自己的 DeepSeek API Key 来生成个性化报告。
   - API Key 只存在浏览器当前会话或本地，不上传到项目服务器。
   - 测试阶段暂不限制生成次数，方便反复验证完整链路。
   - 正式发布前可恢复 localStorage 软限制，但它只能防误点和普通重复生成。
   - 适合内部测试和个人使用。

2. 公开传播模式
   - 前端仍部署到 GitHub Pages。
   - 使用 Cloudflare Worker `ai-test-deepseek-proxy` 作为 API 代理。
   - DeepSeek API Key 放在 Worker Secret / 环境变量里。
   - 用户无需填写 API Key。
   - 使用 Cloudflare D1 数据库 `ai-test-reports` 保存 submissions 和 reports。
   - 代理层负责真实限流、日志、错误处理和成本保护。
   - 这不是传统后端应用，但包含一个轻量 D1 存档数据库，用于公开测试版统计和报告回看。

如果坚持完全没有任何后端或代理，则公开版不能使用站长自己的 DeepSeek Key，只能使用用户自带 Key，或改为本地模板报告。

纯前端限次只能防误点和普通重复生成，不能防恶意用户清缓存、换浏览器、换昵称绕过。测试阶段已临时关闭前端限次；公开测试版先不恢复限流，正式传播前需要在 Worker 代理层按 IP hash、日期、昵称哈希或验证码做真实限流。

公开测试版会保存用户名字/网名、行业、选填职位、选填手机号、答题结果和生成报告，用于改进测试体验和后续反馈。页面必须提醒用户不要填写身份证、详细地址、公司机密等敏感信息；手机号只能选填，不得强制。

## 技术建议

第一版建议使用：

- Vite
- React
- TypeScript
- 静态配置文件保存题目、分值、等级规则、报告 prompt
- GitHub Pages 部署

代码结构建议：

```text
src/
  data/
    questions.ts
    levels.ts
    reportPrompt.ts
  logic/
    scoring.ts
    diagnosis.ts
    deepseek.ts
  components/
    StartScreen.tsx
    QuestionScreen.tsx
    ReportScreen.tsx
    DimensionChart.tsx
  App.tsx
```

## 开发顺序

1. 定稿题目和评分规则。
2. 搭建静态前端页面。
3. 实现本地评分和等级判断。
4. 实现模板报告 fallback。
5. 接入 DeepSeek 生成个性化报告。
6. 公开测试版改为 Cloudflare Worker 代理模式，访问者不填写 API Key。
7. 配置 GitHub Pages 部署。
8. 如需公开传播，再加 Cloudflare Worker 或 Vercel Function 代理。
9. 用不同画像自测，调整题目权重，避免结果虚高。

## 第一版验收标准

- 用户能在手机上顺畅完成 8 道题。
- 用户能得到 Lv.0 到 Lv.10 中的一个等级。
- 用户能看到四维能力分析。
- 用户能得到一份基于自己答案生成的个性化报告。
- 页面可部署到 GitHub Pages。
- 前端代码不泄露 DeepSeek API Key。
- 即使 DeepSeek API 失败，也能显示基础报告，不让用户卡死。
